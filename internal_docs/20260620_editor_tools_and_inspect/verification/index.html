Sprint 1 — Reclaim the header

Goal: the editing view fits the viewport without scrolling; the setup view stays welcoming but tighter.

Removed the redundant in-page <h1> + subtitle in the editing view (the global UHeader already carries the product name) and relocated Undo / New image / mobile Steps into a slim right-aligned toolbar above the grid. Tightened the setup hero padding. The real scroll culprit turned out to be Nuxt UI's UMain min-height filling the viewport below the header and pushing the footer below the fold — fixed by subtracting the footer chrome from UMain's min-height and condensing the UFooter.

Sprint 2 — Crop tool + straightening grid

Goal: the agent can crop for composition/aspect via native RT [Crop], with a rule-of-thirds overlay for the human and stronger straighten guidance.

Added flat crop fields (cropLeft/Top/Width/Height + cropAspect enum) through the full backend chain — types.ts → tools.ts → agent.ts → pp3.ts (RT [Crop]) → executor.ts (Sharp parity) → docs. Added a new presentational GridOverlay.vue (thirds + crop rect) wired into the stage. Crop correctness was proven on the real rawtherapee-cli 5.12 binary, not just the dev Sharp path.

Sprint 3 — One linear graduated filter

Goal: the agent can apply a single linear graduated (ND) filter — the "darken the sky / hold the foreground" local edit — over the corrected global base.

The high-risk sprint: RawTherapee's graduated-filter pp3 keys were not in the verified reference doc, and pp3 fails silently on a wrong key. A Step-0 spike against the 5.12 binary corrected three plan guesses at once — the section is [Gradient] (not [Graduated Filter]), the keys are CenterX/CenterY (capitalized), and Strength = −gradExposure (RT positive darkens). Five flat fields shipped (gradEnabled, gradAngle, gradPosition, gradFeather, gradExposure); none trimmed.

Step 0 spike — RT key verification

Raw evidence from rendering candidate [Gradient] keys through the 5.12 binary. These JPGs are what verified the section name, casing, sign, and the geometry (angle / feather / direction) controls before any of it was wired into the app. The verified keys were recorded in the pp3 reference doc §9.

Sprint 4 — Lightbox as an inspection surface

Goal: zoom/pan + fit/100%, an easy before/after comparison with the original, and keyboard-driven inspection.

A new dependency-free useZoomPan.ts composable drives wheel-zoom-to-cursor, drag pan, two-finger pinch, and double-click fit↔1:1, clamped 0.5×–8×. The lightbox gained zoom controls, a draggable before/after divider that reveals the original (hidden on the original frame), reuse of the Sprint 2 grid overlay, and a full keyboard map. The "100%" readout is true 1:1 derived per-image (naturalWidth / clientWidth), not a fixed multiple.

Key deviations & decisions

  • RT section was [Gradient], not [Graduated Filter]. The Sprint 3.0 spike caught the wrong section name, the CenterX/CenterY casing, and the Strength = −gradExposure sign — all of which would have silently no-op'd. The pp3 reference doc §9 was updated with the verified keys.
  • Crop computed against original dims. The plan feared "rotated output dims" math; the binary showed RT keeps rotated output at original dimensions (scaling content to fit, no wedge), so crop pixels are computed against the original metadata — simpler than expected.
  • The real Sprint-1 scroll culprit was UMain, not the header. Nuxt UI's UMain min-height pushed the footer below the fold; fixed by subtracting footer chrome from its min-height and condensing the UFooter. Grid height retuned 13rem → 13.5rem.
  • Sprint-4 "100%" is true 1:1. Derived per-image so the readout reports actual pixels rather than a mislabeling heuristic multiple.

All deviations were verified against the running app and/or the RT 5.12 binary — none are open risks.

Post-plan enhancement — agent alignment grid

Goal: after a straighten or crop, hand the agent a rule-of-thirds alignment-grid reference image so it can verify the horizon is level / verticals are true — without polluting its color/exposure reads.

Revisited a cut idea, safely. "Baking a grid onto the agent's vision images" was originally rejected because a grid over the primary images corrupts color/exposure reads. This ships it as a separate, additive third image gated to geometry-active iterations: the clean current+original images stay untouched (color/exposure reads stay honest), and the gridded copy is sent ONLY once a straighten or crop is applied. gridReference() (server/utils/pixels.ts) composites the rule-of-thirds SVG via Sharp; decideConfig attaches it as the third image and appends a prompt line telling the model to use it for alignment only. A try/catch degrades gracefully to the prior 2-image behavior on any compositing failure.

End to end

The four capabilities now work as one flow. A slimmer cockpit fits the viewport, so there's room to work; the agent can crop for composition with a rule-of-thirds grid to straighten against; it can lay a graduated filter over the corrected base to darken a sky or lift a foreground; and the human can inspect the result in the lightbox — zoom to 1:1, pan, and slide the before/after divider against the original. Every backend tool renders correctly on the live RawTherapee 5.12 engine, with verified Sharp parity as the fallback.
